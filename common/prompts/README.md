## Code-switch smoke suite

Expanded JSONL (hand-authored, not generated by expand_smoke_suite.py):

```text
glm52_code_switch_suite.expanded.jsonl
```

Counts:

```text
6 language pairs:     en+it, en+zh, en+es, en+fr, en+de, en+pt
1 triple-language:    en+zh+es
3 domains:            coding, math, physics
16 records total
```

Each prompt naturally mixes two languages: part of the request in language A,
the required response in language B — so the model must parse both to answer.
`language` is a multi-segment label (e.g. `en+it`); `script` is `Mixed`.

Run a code-switch batched study (model loads once):

```bash
SUITE="prompts/tracing/glm52_code_switch_suite.expanded.jsonl" \
LANGS="" DOMAINS="" ONE_PER_COMBO=0 N_PRED=16 \
TRACE_OUT="traces/batch/code_switch" \
REPORT_MD="reports/glm52_code_switch_report.md" \
REPORT_JSON="reports/glm52_code_switch_summary.json" \
bash scripts/tracing/run_trace_suite_batched.sh
# NOTE: empty LANGS="" now means "no filter, keep all" (was previously falling
# back to the default 7-language list and silently dropping code-switch labels).
```

Purpose:

- satisfy Story 7's code-switching AC ("can be labeled with multiple languages,
  such as `en+it` or `en+zh`")
- verify the analyzer treats `language` as an opaque string label and that
  multi-segment languages (`en+zh+es`) aggregate naturally
- study whether code-switch routing (a) raises router entropy and (b) unions
  the two component languages' expert repertoires

Caveat: 16 prompts is a SMALL N; per-(pair, domain) claims need replication
across multiple test variants before being trusted (same lesson as the 49→161
scaled study for the monolingual suite). Treat the code-switch study as
framework-validation + first-look data.
